The design tier is where the reasoning lives: why the rib is built the way it is,
and what it depends on from the harness. Each record states a decision, the
alternative it rejected, and why. This is the home for design narrative the source
comment policy keeps out of the code.

## What Chamber needs from the harness

Chamber adds no harness internals of its own. It attaches only through the
`RibContext` seams the harness hands it, and it fails closed when a load-bearing
seam is absent rather than degrading silently.

| Seam | What the rib does with it | When absent |
|---|---|---|
| `runAgentTurn` | run a room, lens, or briefing turn | the room driver and room tools are not built; room actions fail closed |
| `getSnapshotManager` | publish boards to `rib:chamber:*` keys | lens and room publishing are unavailable |
| `registerRegion` | add a lens panel at runtime | lenses are unavailable |
| `getDataDir` | a writable per-rib home for Minds, rooms, and lenses | falls back to a resolved per-rib data directory |
| `getProjects` | resolve a targeted project's root path for coding room turns | coding rooms reject a targeted project id at start (fail closed); project targeting is unavailable |

A sixth seam, `refreshWorkflow`, lets an index republish on demand instead of
waiting on its cadence; without it the rib falls back to cadence-only refresh. The
rib also contributes workflows whose nodes bind a snapshot key and a fail-closed
`validate`, so a producer's output is gated before it reaches a trusted renderer.

The records below assume that model. For what a rib is in general, and the full
context contract, see the
[Keelson rib contract](https://danielscholl.github.io/keelson/docs/reference/rib-contract/).
